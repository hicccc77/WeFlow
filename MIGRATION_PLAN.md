# WeFlow Migration Plan: One Communication Assistant → TypeScript Rewrite

> **Strategy: B Plan** — TypeScript native rewrite, not Python subprocess.
> All intelligence features from One's Python backend are rewritten as TypeScript services
> running natively in Electron's main process.

---

## Strategy Overview

The One project's communication assistant (reply coach, social graph, PUA detection, daily briefing, etc.) is implemented in Python (~130K+ lines across multiple modules). Rather than spawning Python subprocesses from Electron (A plan), we rewrite each service in TypeScript (B plan) for:

- **Zero runtime dependency** — no Python install required on user machines
- **Tighter IPC** — direct function calls instead of HTTP/subprocess overhead
- **Type safety** — shared interfaces enforce contracts between frontend and backend
- **Single build pipeline** — one `npm run build` produces everything

## Interface Contracts

These TypeScript types are the **binding contracts** between roles. Any change requires agreement from all consuming roles.

### Media Layer (Service Dev A → Service Dev B)

```typescript
/** Output of mediaContextService for a single media attachment */
interface MediaContext {
  /** Original message type: image, voice, video, article, forward */
  type: 'image' | 'voice' | 'video' | 'article' | 'forward'
  /** Human-readable description generated by LLM or transcription */
  description: string
  /** Original file path or URL (for cache key) */
  sourcePath: string
  /** Processing duration in ms (for performance tracking) */
  processingMs: number
  /** Whether this came from cache */
  cached: boolean
  /** Raw base64 data (only for images, only when needed by Vision LLM) */
  base64Data?: string
  /** Confidence score 0-1 for the description quality */
  confidence: number
}
```

### Analysis Layer (Service Dev B → Frontend Dev)

```typescript
/** Result of analyzing an incoming message */
interface AnalysisResult {
  /** Whether the message requires multi-round discussion before reply */
  isComplex: boolean
  /** Why the message was classified as complex/simple */
  reason: string
  /** Guided questions to help user think through their reply */
  guideQuestions: string[]
  /** Direct reply suggestions (only for non-complex messages) */
  suggestions?: ReplySuggestion[]
  /** Discussion session ID (only for complex messages) */
  discussionId?: string
}

/** A single reply suggestion with metadata */
interface ReplySuggestion {
  /** The suggested reply text */
  content: string
  /** Why this reply was suggested */
  reason: string
  /** Communication style of this suggestion */
  style: 'safe' | 'warm' | 'firm'
  /** Model confidence 0-1 */
  confidence: number
}

/** A single round in a multi-round discussion */
interface DiscussionRound {
  /** Who sent this message */
  role: 'user' | 'assistant'
  /** Message content */
  content: string
  /** ISO 8601 timestamp */
  timestamp: string
}
```

### IPC Contract (Frontend Dev ↔ All Backend Roles)

```typescript
/** 
 * IPC channel definitions.
 * Frontend calls these via window.ipc.invoke(channel, ...args)
 * Main process registers handlers in main.ts
 */
interface AssistantIPC {
  /** Analyze an incoming message, return complexity + suggestions or discussion */
  'intel:analyzeMessage': (
    contact: string,
    message: string,
    selectedContext?: string[]
  ) => Promise<AnalysisResult>

  /** Continue a multi-round discussion about how to reply */
  'intel:discuss': (
    contact: string,
    message: string,
    userInput: string,
    discussionId?: string
  ) => Promise<{
    analysis: string
    followup?: string
    discussionId: string
  }>

  /** After discussion is complete, generate reply suggestions */
  'intel:discussReply': (
    discussionId: string
  ) => Promise<ReplySuggestion[]>

  /** Get chat context for a contact (paginated) */
  'intel:getContext': (
    contact: string,
    offset?: number,
    limit?: number
  ) => Promise<{
    messages: any[]
    hasMore: boolean
    total: number
    isGroup: boolean
  }>

  /** Star a contact for priority treatment */
  'intel:starContact': (contactId: string) => Promise<void>

  /** Ignore a contact (hide from assistant) */
  'intel:ignoreContact': (contactId: string) => Promise<void>

  /** Get a coach log entry by ID (for debug panel) */
  'intel:getCoachLog': (logId: number) => Promise<CoachLogEntry>
}
```

## Known Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| One `reply_coach.py` ~130K lines | Large rewrite scope, easy to miss edge cases | Port by feature priority: core pipeline first, edge cases in Phase 3 |
| LLM API cost during development | Testing burns API quota rapidly | Use Claude Code CLI zero-cost mode for development; mock LLM in unit tests |
| WCDB schema changes between WeChat versions | WeChat updates break database reads | Abstract all WCDB queries behind a version-aware adapter layer |
| Image analysis performance | base64 transfer to LLM is slow for large images | Resize to max 1024px + JPEG compress before sending; set 30s timeout |
| Electron main process blocking | CPU-intensive analysis blocks UI rendering | Run heavy tasks (media processing, LLM calls) in Worker threads |
| Prompt quality regression | TypeScript prompts may not match Python originals | A/B test against One's output; keep prompts in separate `.txt` files for easy editing |
| Voice transcription accuracy | Local inference may be lower quality than cloud | Fall back to LLM-based transcription if local confidence < threshold |
| Cache invalidation correctness | Stale cache serves wrong suggestions | Hash full context (not just message) as cache key; TTL max 1 hour |
